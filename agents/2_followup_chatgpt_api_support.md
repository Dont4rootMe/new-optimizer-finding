# Follow-up: добавление ChatGPT API support и переключение на pro thinking модель

## Какой был контекст прямо перед изменением

На момент этого изменения в репозитории уже существовал модуль `evolve`, который умел:

- генерировать кандидатов оптимизаторов через `src/evolve/generator.py`;
- обращаться к OpenAI Python SDK через `from openai import OpenAI`;
- использовать `Responses API` как основной путь и `chat.completions` как fallback;
- работать с `provider=openai`;
- сохранять `llm_request.json` и `llm_response.json`;
- запускать эволюционный пайплайн через `python -m src.main mode=evolve`.

При этом у LLM-конфига были два практических ограничения:

- по умолчанию в конфиге стояла не pro-thinking модель, а более старый дефолт;
- не было явного режима `provider=chatgpt` как пользовательского способа сказать: "используй ChatGPT API / OpenAI SDK c актуальной мощной моделью";
- не было удобного алиаса вида `latest_pro_thinking`, чтобы не хардкодить идентификатор модели в каждом запуске.

Запрос был таким: добавить поддержку ChatGPT API и использовать самую последнюю pro мощную thinking модель как дефолт.

## Что именно я сделал

### 1. Обновил генератор кандидатов

Файл: `src/evolve/generator.py`

В генераторе были добавлены:

- константа:
  - `LATEST_PRO_THINKING_MODEL = "gpt-5.4-pro"`
- алиасы моделей:
  - `latest_pro_thinking`
  - `latest_pro`
  - `chatgpt_pro_thinking`

Это было сделано для того, чтобы:

- иметь один централизованный дефолт для "самой мощной pro thinking" модели;
- не размазывать строковый идентификатор модели по конфигам и ручным override-командам;
- поддержать удобный UX для CLI, когда пользователь может писать не только точный model id, но и понятный alias.

Также в `OptimizerGenerator.__init__` я добавил:

- `self.provider = str(self.llm_cfg.provider).lower()`
- `self.model_name = self._resolve_model_name()`

После этого появился отдельный метод `_resolve_model_name()`, который:

- берёт `evolver.llm.model` из Hydra-конфига;
- если поле пустое, возвращает `gpt-5.4-pro`;
- если поле содержит alias, разворачивает его в реальный model id.

### 2. Добавил явную поддержку `provider=chatgpt`

В методе `_call_openai(...)` логика была расширена так, чтобы провайдером можно было задавать:

- `openai`
- `chatgpt`
- `mock`

Если приходит что-то иное, теперь выбрасывается понятная ошибка:

- `Unsupported LLM provider '<...>'. Use one of: openai, chatgpt, mock.`

Здесь важно: я не вводил отдельный новый SDK для "chatgpt". Реализация по-прежнему использует официальный OpenAI Python SDK, то есть:

```python
from openai import OpenAI
```

А значение `provider=chatgpt` сделано как человеко-понятный конфиг-алиас поверх того же рабочего механизма. Это осознанное решение, потому что в коде фактически нужен не другой транспорт, а другой пользовательский режим конфигурирования.

### 3. Переключил запросы на resolved model name

Ранее запросы использовали напрямую строку из `self.llm_cfg.model`.

Я заменил это на `self.model_name` в двух местах:

- при вызове `client.responses.create(...)`;
- при fallback-вызове `client.chat.completions.create(...)`.

Это нужно, чтобы:

- alias разрешался один раз при инициализации генератора;
- downstream-код всегда работал с уже нормализованным идентификатором модели;
- request payload и сохранённые метаданные были согласованы между собой.

### 4. Добавил раннюю проверку `OPENAI_API_KEY`

В `_call_openai(...)` добавлена явная проверка:

- если `OPENAI_API_KEY` не установлен, выбрасывается понятный `RuntimeError`.

Сообщение:

- `OPENAI_API_KEY is not set. Export OPENAI_API_KEY before running evolve.`

Это было сделано специально как UX-улучшение. Без этой проверки пользователь получал бы менее точную ошибку уже изнутри SDK. С ранней валидацией причина проблемы становится очевидной сразу.

### 5. Сохранил существующую поддержку `OPENAI_BASE_URL`

Я не менял уже рабочую логику, которая читает:

- `OPENAI_BASE_URL`

и прокидывает его в `OpenAI(**client_kwargs)`.

То есть после изменения система поддерживает:

- обычный OpenAI API через `OPENAI_API_KEY`;
- кастомный совместимый base URL, если это нужно инфраструктурно.

### 6. Обновил mock-ветку так, чтобы она тоже использовала resolved model name

В коде генерации мок-кандидата поле `model` в `request_payload` теперь тоже пишет `self.model_name`, а не сырое значение из конфига.

Это мелкое, но важное выравнивание:

- mock и real provider теперь отражают один и тот же resolved model contract;
- легче сопоставлять поведение между тестовым и реальным режимами.

### 7. Поменял дефолтный конфиг evolver

Файл: `conf/evolver/default.yaml`

Я обновил блок `llm` следующим образом:

- `provider: chatgpt`
- `model: gpt-5.4-pro`
- `temperature: 0.5`
- `reasoning_effort: xhigh`

Смысл этих изменений:

- сделать именно ChatGPT/OpenAI path дефолтным способом генерации;
- по умолчанию использовать pro-thinking модель;
- опустить `temperature` до `0.5`, чтобы уменьшить шум при генерации optimizer-кандидатов;
- явно включить более тяжёлое reasoning через `xhigh`.

Это изменение было сделано в духе запроса: не просто поддержать API, а сразу сделать "мощный thinking" дефолтом репозитория.

### 8. Обновил README

Файл: `README.md`

В документации я добавил:

- пример запуска с `OPENAI_API_KEY`;
- фиксацию новых дефолтов:
  - `provider=chatgpt`
  - `model=gpt-5.4-pro`
  - `reasoning_effort=xhigh`
- пример override через alias:

```bash
python -m src.main mode=evolve evolver.llm.model=latest_pro_thinking
```

Это было нужно, чтобы после изменения пользователь мог:

- увидеть новый способ запуска прямо из README;
- понять, какой модельный дефолт теперь используется;
- быстро переопределить модель без чтения кода.

## Что я намеренно не менял

Я не переписывал весь транспорт вызова моделей, потому что:

- код уже был правильно построен вокруг официального OpenAI SDK;
- основной вызов через `responses.create(...)` уже соответствовал нужной архитектуре;
- отдельной "chatgpt-only" ветки на уровне SDK не требовалось.

Я также не менял остальную эволюционную логику:

- оркестратор,
- Neyman allocation,
- GPU pool,
- scoring,
- runner validate-mode.

Запрос касался именно LLM-доступа и выбора модельного дефолта, поэтому я ограничил изменение этим контуром.

## Какой получился итоговый runtime behavior

После изменений сценарий работы стал таким:

1. Пользователь запускает `mode=evolve`.
2. Hydra берёт `conf/evolver/default.yaml`.
3. По умолчанию выбирается:
   - `provider=chatgpt`
   - `model=gpt-5.4-pro`
   - `reasoning_effort=xhigh`
4. `OptimizerGenerator` резолвит model alias в реальный model id.
5. При отсутствии `OPENAI_API_KEY` падение происходит рано и с понятным сообщением.
6. При наличии ключа генератор идёт в:
   - `client.responses.create(...)`
   - если текст не извлёкся и fallback разрешён, то в `client.chat.completions.create(...)`
7. Request/response сохраняются на диск как и раньше.

## Что было проверено после правок

После изменения я прогнал проверку на компилируемость Python-кода через `compileall`.

Причина, почему я использовал именно это:

- это быстрый smoke-check на синтаксическую целостность;
- в текущем окружении на тот момент не был гарантирован полноценный запуск всех тестов с внешними зависимостями.

Проверка компиляции прошла успешно.
