# STATE_REPRESENTATION
This section defines what internal problem state, derived state, decomposition, or working representation the heuristic uses before planning happens. Put genes here that describe parsed grid structures, robot summaries, target summaries, region or zone labels, congestion or pressure maps, wall abstractions, route state, distance or potential fields, or other state that later sections consume. This section is about representation, not the global strategy itself and not local repair. Coordinate with `MACRO_STRATEGY` only when the represented quantities are exactly what that section frames as the global hypothesis; do not duplicate strategy decisions here. State definitions must be well-posed: if the genes invoke a "congestion score" or a "region label", the representation section must define how it is computed from the input, not just name it. Plain language, mathematical descriptions, pseudocode, and concrete Python fragments are all first-class here. When this section defines a substantive derived quantity (a distance map, a congestion score, a region label), the default is to include a short pseudocode or Python block that fixes how it is computed; pure prose is reserved for cases where the rule is naturally one sentence. This section must not contain a full end-to-end `solve_case` implementation.

# MACRO_STRATEGY
This section defines the high-level global plan or decomposition strategy for the heuristic. Put genes here that describe board partitioning, grouping philosophy, routing phases, wall-planning strategy, global flow assumptions, staged release of robots, or other organism-level strategy. This section is about the global hypothesis: what the organism believes will reduce the absolute score, why it should work, and what families of inputs it targets. It is not the place for detailed construction steps and not the place for local repair. Macro strategy should usually express one coherent global hypothesis, not multiple incompatible alternatives or a list of fallbacks; pick one and commit to it. This is the one section that must remain plain language plus mathematical or structured-planning prose; do not place Python code or detailed pseudocode here. Save executable detail for the construction, repair, and code-sketch sections that follow. Like every required subsection, every line of MACRO_STRATEGY must be a `- ` bullet; paragraph prose without leading `- ` is a hard parse error. Multi-sentence ideas continue on the next line indented by exactly two spaces, never as a fresh unbulleted line.

# CONSTRUCTION_POLICY
This section defines how the actual contest output is built and how choices are ranked while it is being built. Put genes here that describe added-wall construction (which cells get walls, in what order, under what local rule), group assignment construction (how many groups, how robots map to them, what makes the assignment respect the macro strategy), route or command construction (how individual or group commands are emitted to make progress), the priority or ranking logic that decides what gets processed next during construction, and how staging or interleaving across these blocks works. Ranking choices that used to live in a separate priority section now belong here: a robot, target, group, route, or wall-cell ordering must be deterministic given the represented state, expressible as a small number of compact comparable scores or rules, with explicit tie-breaking. Construction must respect the explicit output shape (`N` lines vertical, `N - 1` lines horizontal, `K` lines of group ids, then operations). Construction must respect the operation budget `T <= K * N * N`. Inline parameter choices, thresholds, schedules, ratios, and group counts as a function of `K` directly here at the place they are used, expressed in terms of the input parameters `N` and `K` rather than as raw numbers. Plain language, mathematical descriptions, pseudocode, and Python fragments are all first-class here. The default for any substantive construction or ranking rule is to include a short pseudocode or Python block that pins the procedure (the comparator body, the wall-cell ordering rule, the per-round move-selection loop); reserve pure prose for cases where the mechanism is naturally a single-sentence rule. Do not push every code fragment out into `OPTIONAL_CODE_SKETCH` if it actually belongs to construction. This section must not contain a full end-to-end `solve_case` implementation.

# LOCAL_REPAIR_POLICY
This section defines deterministic corrective logic when local defects, conflicts, congestion, deadlocks, blocked routes, or inefficient group commands are detected during or after construction, together with the outer control logic that sequences phases and decides when the algorithm stops. Put genes here that describe rerouting, command trimming, collision avoidance, local wall adjustment, delayed release of a stuck robot, fallback to individual moves, single-step backtracking, or other localized repair mechanisms; and put genes here that describe stage sequencing (for example, build walls before forming groups, or interleave wall-and-group co-design), acceptance rules, convergence checks, iteration budgets, and escalation from coarse to fine repair. Control should remain minimal and decisive; avoid baroque multi-phase orchestration unless the mechanism truly requires it. The repair-and-control logic must respect the same operation budget as construction (`T <= K * N * N`); a repair that would exceed the budget must be guarded or capped, and there must be a guaranteed termination path. Inline parameter choices, thresholds, and iteration caps directly here at the place they are used, expressed in terms of `N` and `K`. Plain language, mathematical descriptions, pseudocode, and Python fragments are all first-class here. The default for any substantive repair operator or control loop is to include a short pseudocode or Python block that pins the mechanism (the rerouting trigger, the stuck-robot escalation body, the phase-transition condition); reserve pure prose for cases where the rule is naturally one sentence. Do not push every code fragment out into `OPTIONAL_CODE_SKETCH` if it actually belongs to repair or control. This section must not contain a full end-to-end `solve_case` implementation.

# OPTIONAL_CODE_SKETCH
This section is reserved specifically for code-shaped detail that does NOT belong to one specific earlier section: cross-section helpers, reusable utility functions used by both construction and repair, named comparator definitions referenced from multiple places, or scratch worked examples. If a code fragment is local to one mechanism, place it inside the section that owns that mechanism (`STATE_REPRESENTATION`, `CONSTRUCTION_POLICY`, or `LOCAL_REPAIR_POLICY`) rather than in here — using this section as the only home for code is a regression. This section must never contain a full `implementation.py`, must never define `solve_case` end-to-end, must never redefine the whole heuristic from scratch, and must never replace the higher-level genetic descriptions in the earlier sections. Use `- None.` whenever no genuinely cross-section code helper is needed — purely textual designs are fully valid and a `- None.` entry is not penalized.
