# STATE_REPRESENTATION
This section defines what internal problem state, derived state, decomposition, or working representation the heuristic uses before planning happens. Put genes here that describe parsed grid structures, robot summaries, target summaries, region or zone labels, congestion or pressure maps, wall abstractions, route state, distance or potential fields, or other state that later sections consume. This section is about representation, not the global strategy itself and not local repair. Coordinate with `MACRO_STRATEGY` and `PRIORITY_MODEL` only when the represented quantities are exactly what those sections rank or schedule on; do not duplicate strategy decisions here. Plain language, mathematical descriptions, pseudocode-like descriptions, and named data structures are allowed here. State definitions must be well-posed: if the genes invoke a "congestion score" or a "region label", the representation section must define how it is computed from the input, not just name it. Full Python code is not allowed here.

# MACRO_STRATEGY
This section defines the high-level global plan or decomposition strategy for the heuristic. Put genes here that describe board partitioning, grouping philosophy, routing phases, wall-planning strategy, global flow assumptions, staged release of robots, or other organism-level strategy. This section is about the global hypothesis: what the organism believes will reduce the absolute score, why it should work, and what families of inputs it targets. It is not the place for detailed construction steps and not the place for local repair. Plain language, mathematical descriptions, and structured planning descriptions are allowed here. Macro strategy should usually express one coherent global hypothesis, not multiple incompatible alternatives or a list of fallbacks; pick one and commit to it. Full Python code is not allowed here.

# CONSTRUCTION_POLICY
This section defines how the actual contest output is built: the added wall pattern, the group assignment, and the operation sequence. Put genes here that describe added-wall construction (which cells get walls, in what order, under what local rule), group assignment construction (how many groups, how robots map to them, what makes the assignment respect the macro strategy), route or command construction (how individual or group commands are emitted to make progress), and how staging or interleaving across these three blocks works. This section is about building the candidate solution, not ranking choices and not repairing defects after detection. Plain language, mathematical descriptions, pseudocode-like descriptions, and short procedural sketches are allowed here. Construction must respect the explicit output shape (`N` lines vertical, `N - 1` lines horizontal, `K` lines of group ids, then operations). Construction must respect the operation budget `T <= K * N * N`. Full Python code is not allowed here.

# PRIORITY_MODEL
This section defines the ranking or choice logic used to decide what gets processed next during construction or repair. Put genes here that describe robot priority, target priority, group release order, congestion-based ranking, route-choice ordering, wall-cell ordering, or other deterministic selection criteria. This section is about priority and ranking, not the mechanics of construction or repair themselves. The ranking must be deterministic given the represented state: it must define a strict order or a deterministic tie-break, not a vague preference. Plain language, mathematical descriptions, and structured numeric reasoning are allowed here. Priority criteria should be expressible as a small number of compact comparable scores or rules, not as many loosely interacting heuristics. Full Python code is not allowed here.

# LOCAL_REPAIR_POLICY
This section defines deterministic corrective logic used when local defects, conflicts, congestion, deadlocks, blocked routes, or inefficient group commands are detected during or after construction. Put genes here that describe rerouting, command trimming, collision avoidance, local wall adjustment, delayed release of a stuck robot, fallback to individual moves, single-step backtracking, or other localized repair mechanisms. This section may carry slightly more operational detail than the earlier sections, but it must still describe a repair mechanism, not a full end-to-end solver. The repair policy must respect the same operation budget as construction (`T <= K * N * N`); a repair that would exceed the budget must be guarded or capped. Plain language, mathematical descriptions, pseudocode-like descriptions, numeric choices, and if truly local to repair only, short local code-like sketches are allowed here. Do not place a full end-to-end implementation in this section.

# CONTROL_POLICY
This section defines the outer control logic of the organism: how phases are ordered, when construction yields to repair, when the algorithm switches modes, when it stops, and what counts as no further useful progress. Put genes here that describe stage sequencing (for example, build walls before forming groups, or interleave wall-and-group co-design), acceptance rules, convergence checks, iteration budgets, escalation from coarse to fine repair, or similar global control decisions. This section is about algorithm-level control flow, not local construction details and not low-level repair mechanics. Control should remain minimal and decisive; avoid baroque multi-phase orchestration unless the mechanism truly requires it. The control policy must define a guaranteed termination path that does not exceed the operation budget. Plain language, mathematical descriptions, pseudocode-like descriptions, and numeric thresholds are allowed here. Full Python code is not allowed here.

# PARAMETERS
This section stores explicit parameter choices, schedules, thresholds, relative coefficients, limits, ratios, group counts as a function of `K`, wall-density targets as a function of `N`, operation budgets as a fraction of `K * N * N`, and other quantitative settings used by the other sections. Put genes here that make the organism's quantitative choices specific, expressed in terms of the input parameters `N` and `K` rather than as raw numbers. Parameter choices may support any earlier section; they are not decorative comments and should not read like a late appendix. This section may contain numbers, formulas, inequalities, ranges, ratios, ordering constraints, or named parameter bundles in plain text. Parameter blocks should stay compact and grouped; do not explode the organism into many loosely connected constants. This section must not introduce a brand-new heuristic mechanism that properly belongs in another section. Full Python code is not allowed here.

This section is for local inherited procedural detail, not for a generic emergency feasibility reset, not for a hidden global algorithm override, and not for a universal terminal safety clamp. This section must never contain a full `implementation.py`, must never define `solve_case` end-to-end, must never redefine the whole heuristic, and must never replace the higher-level genetic descriptions in the earlier sections.
