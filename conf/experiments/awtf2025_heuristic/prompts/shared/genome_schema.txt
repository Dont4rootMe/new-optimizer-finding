# STATE_REPRESENTATION
This section defines what internal problem state, derived state, decomposition, or working representation the heuristic uses before planning happens. Put genes here that describe parsed grid structures, robot summaries, target summaries, region or zone labels, congestion or pressure maps, wall abstractions, route state, distance or potential fields, or other state that later sections consume. This section is about representation, not the global strategy itself and not local repair. Coordinate with `MACRO_STRATEGY` only when the represented quantities are exactly what that section frames as the global hypothesis; do not duplicate strategy decisions here. State definitions must be well-posed: if the genes invoke a "congestion score" or a "region label", the representation section must define how it is computed from the input, not just name it. Plain language, mathematical descriptions, pseudocode, and concrete Python fragments are all welcome here whenever they sharpen the definition; a textual description is equally valid when the idea is naturally expressed as prose. This section must not contain a full end-to-end `solve_case` implementation.

# MACRO_STRATEGY
This section defines the high-level global plan or decomposition strategy for the heuristic. Put genes here that describe board partitioning, grouping philosophy, routing phases, wall-planning strategy, global flow assumptions, staged release of robots, or other organism-level strategy. This section is about the global hypothesis: what the organism believes will reduce the absolute score, why it should work, and what families of inputs it targets. It is not the place for detailed construction steps and not the place for local repair. Macro strategy should usually express one coherent global hypothesis, not multiple incompatible alternatives or a list of fallbacks; pick one and commit to it. This is the one section that must remain plain language plus mathematical or structured-planning prose; do not place Python code or detailed pseudocode here. Save executable detail for the construction, repair, and code-sketch sections that follow.

# CONSTRUCTION_POLICY
This section defines how the actual contest output is built and how choices are ranked while it is being built. Put genes here that describe added-wall construction (which cells get walls, in what order, under what local rule), group assignment construction (how many groups, how robots map to them, what makes the assignment respect the macro strategy), route or command construction (how individual or group commands are emitted to make progress), the priority or ranking logic that decides what gets processed next during construction, and how staging or interleaving across these blocks works. Ranking choices that used to live in a separate priority section now belong here: a robot, target, group, route, or wall-cell ordering must be deterministic given the represented state, expressible as a small number of compact comparable scores or rules, with explicit tie-breaking. Construction must respect the explicit output shape (`N` lines vertical, `N - 1` lines horizontal, `K` lines of group ids, then operations). Construction must respect the operation budget `T <= K * N * N`. Inline parameter choices, thresholds, schedules, ratios, and group counts as a function of `K` directly here at the place they are used, expressed in terms of the input parameters `N` and `K` rather than as raw numbers. Plain language, mathematical descriptions, pseudocode, and Python fragments are all welcome here whenever they sharpen the mechanism; do not artificially water a clear procedural idea down to prose when a few lines of pseudocode or Python would express it better. Equally, do not pad with code when a textual rule is naturally compact. This section must not contain a full end-to-end `solve_case` implementation.

# LOCAL_REPAIR_POLICY
This section defines deterministic corrective logic when local defects, conflicts, congestion, deadlocks, blocked routes, or inefficient group commands are detected during or after construction, together with the outer control logic that sequences phases and decides when the algorithm stops. Put genes here that describe rerouting, command trimming, collision avoidance, local wall adjustment, delayed release of a stuck robot, fallback to individual moves, single-step backtracking, or other localized repair mechanisms; and put genes here that describe stage sequencing (for example, build walls before forming groups, or interleave wall-and-group co-design), acceptance rules, convergence checks, iteration budgets, and escalation from coarse to fine repair. Control should remain minimal and decisive; avoid baroque multi-phase orchestration unless the mechanism truly requires it. The repair-and-control logic must respect the same operation budget as construction (`T <= K * N * N`); a repair that would exceed the budget must be guarded or capped, and there must be a guaranteed termination path. Inline parameter choices, thresholds, and iteration caps directly here at the place they are used, expressed in terms of `N` and `K`. Plain language, mathematical descriptions, pseudocode, and Python fragments are all welcome here whenever they sharpen the mechanism. This section must not contain a full end-to-end `solve_case` implementation.

# OPTIONAL_CODE_SKETCH
This section is optional in spirit but must always exist in the schema. It is reserved for executable or near-executable sketches that help preserve heritable procedural detail: pseudocode, Python-like fragments, or small reusable helper definitions that several earlier sections rely on. Use this section freely whenever a concrete code shape is the clearest way to lock in an idea — reusable helpers, comparator functions, inner loops of a specific operator, named utility functions, or small worked examples are all valid. Code here may span more than one mechanism if the same fragment is genuinely shared across construction and repair. This section must never contain a full `implementation.py`, must never define `solve_case` end-to-end, must never redefine the whole heuristic from scratch, and must never replace the higher-level genetic descriptions in the earlier sections. Use `- None.` only when no code-shaped detail is helpful — purely textual designs are fully valid and a `- None.` entry is not penalized.
