TASK: Graph Colouring

Problem description:
Given a graph in DIMACS format with vertices, edges, and an adjacency list, assign a positive integer colour to each vertex (1..n) so that no two adjacent vertices share the same colour. This is the classic graph (vertex) colouring problem: find a proper colouring of an undirected graph G = (V, E) — a labelling of the vertices such that every edge connects two vertices of different colours — that uses as few distinct colours as possible. The minimum achievable number is the graph's chromatic number.

solve(**kwargs) contract for this task:
- INPUT keys (each instance passes these by keyword to solve):
  - `n` (int): the number of vertices. Vertices are labelled 1..n.
  - `edges` (list): the undirected edges as (u, v) integer tuples, e.g. [(1, 2), (2, 3), ...].
  - `adjacency` (dict): maps each vertex (1..n) to the set of its adjacent vertices, e.g. {1: {2, 5}, 2: {1, 3}, ...}. This is the same information as `edges`, pre-indexed per vertex for convenience.
- REQUIRED OUTPUT: return a FLAT dict that maps every vertex_id (1..n) to a positive integer colour, e.g. {1: 1, 2: 2, 3: 1, ...}. IMPORTANT: the dict's keys are the vertex ids themselves and its values are the assigned colours — do NOT wrap the assignment under any key (it is a plain {vertex: colour} mapping, not {"colors": {...}} or {"solution": {...}}). Every vertex from 1 to n must appear exactly once as a key, and every colour must be an integer >= 1.

Feasibility (a colouring is rejected — scores 0 — unless all hold):
- Every vertex from 1 to n is assigned a colour (no vertex left unassigned).
- Every assigned colour is a positive integer (>= 1).
- No edge is monochromatic: for every edge (u, v), the colour of u differs from the colour of v. A single same-coloured adjacent pair (a conflict) invalidates the whole solution and yields no score.

Objective and scoring:
- The raw score of a valid colouring is k, the number of distinct colours it uses. LOWER k is better (this is a minimization problem).
- The evaluator normalizes each instance's score against a known optimal number of colours (optimal / k), so the normalized score is in (0, 1] and HIGHER is better; an optimal colouring scores 1.0.
- Any conflict (an edge whose endpoints share a colour) or any unassigned vertex makes the solution invalid and it earns no score, so always return a complete, proper colouring before trying to reduce the colour count.

Reference baseline:
- Giving every vertex its own distinct colour ({v: v for v in range(1, n + 1)}) is always a VALID proper colouring (no two vertices share a colour, so no conflict) and scores > 0, but it uses n colours — the worst possible. It is only a feasibility floor: improve on it by reusing colours across non-adjacent vertices (e.g. a greedy first-fit colouring over a good vertex ordering, optionally followed by local improvement) to drive the colour count down toward the chromatic number.
