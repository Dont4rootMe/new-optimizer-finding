# CONSTRUCTION
This section defines how the organism builds an initial candidate solution for the supplied instance before any later improvement or repair. Put genes here that describe the constructive principle: greedy versus randomized-greedy construction, the ordering or priority rule used to commit decisions, how a seed/reference solution (when the instance supplies one) is exploited, decomposition of the instance into subproblems, or any other principle that materially shapes the starting solution quality. Content here should stay focused on how the first feasible (or near-feasible) solution is produced, not on how it is later improved and not on how violations are repaired later. This section may need to be coordinated with `FEASIBILITY_MODEL` and `REPAIR_POLICY` when one hypothesis couples construction order, attainable feasibility, and repair behavior. Plain language, mathematical descriptions, and pseudocode-like procedural descriptions are allowed here. Prefer a low-parameter constructive rule over a richly over-specified pipeline unless the extra structure is truly necessary for score. Full Python code is not allowed here.

# SEARCH_STRATEGY
This section defines the overall improvement strategy the organism uses to drive the objective up (or the cost down) after construction. Put genes here that describe the score-driving search regime: local search, best-improvement versus first-improvement acceptance, metaheuristic family (simulated annealing, tabu search, GRASP, large-neighborhood search, evolutionary, etc.), restart policy, and how search effort is allocated across the instance. This section is specifically about the macro improvement dynamics, not the concrete move set (that belongs in `NEIGHBORHOOD`) and not the constraint-violation accounting (that belongs in `FEASIBILITY_MODEL`). It should not collapse into a generic slogan like "improve the solution"; it should describe the actual regime that could move score within the per-instance time budget. Plain language, mathematical descriptions, pseudocode-like descriptions, and numeric schedules are allowed here. Prefer one clear search regime over multiple nested sub-strategies unless their interaction is essential. Full Python code is not allowed here.

# NEIGHBORHOOD
This section defines the concrete move operators and neighborhood structure the search uses to transform one candidate into another. Put genes here that describe the moves (swap, insertion, 2-opt / k-opt, relocate, ruin-and-recreate, flip, segment reversal, block move, etc.), how candidate moves are generated and screened, any move-ordering or candidate-list pruning, and how incremental cost is evaluated for a move. This section is about the move set and how it is explored, not about the high-level strategy that calls it and not about constraint definitions. The neighborhood should be expressed as a compact, well-defined set of operators rather than many loosely coupled exceptions. Plain language, mathematical descriptions, and structured numeric reasoning are allowed here. Full Python code is not allowed here.

# FEASIBILITY_MODEL
This section defines what the organism treats as a feasible solution and how it accounts for constraint violations during search. Put genes here that describe the hard constraints of the instance as the organism models them, how violations are detected and quantified, whether infeasible solutions are forbidden or penalized, how a penalty or violation measure is defined, which violations matter most, and what summary quantities are maintained during the algorithm. This section is about defining feasibility and violation measures, not about how violations are fixed. Any penalty or violation measure must be mathematically well-posed and aligned with how repair and search consume it. Plain language, mathematical descriptions, and pseudocode-like descriptions are allowed here. Full Python code is not allowed here.

# REPAIR_POLICY
This section defines how the organism restores feasibility once a constructed or mutated solution violates a constraint. Put genes here that describe repair operators such as ejection/insertion, constraint-directed reassignment, worst-violation-first correction, sequential or propagated repair, role-aware repair, or other deterministic feasibility-restoration logic that tries to preserve objective quality. This section should preserve solution quality whenever possible; a wholesale reset to a trivial feasible solution is normally not a repair policy and should not be the default cleanup mechanism unless the organism's actual hypothesis is built around restart-based correction. This section may contain local procedural detail and may be more operational than the earlier sections, but it must still describe a repair mechanism rather than a full end-to-end implementation. Plain language, mathematical descriptions, pseudocode-like descriptions, numeric choices, and (only when truly local to repair) short local code-like sketches are allowed here. Do not place a full algorithm implementation in this section.

# CONTROL_POLICY
This section defines the outer control logic of the organism: how phases are ordered, when construction hands off to improvement, when search restarts or intensifies versus diversifies, how the per-instance time or iteration budget is spent, what acceptance and stopping rules apply, and what counts as no more profitable feasible updates. Put genes here that describe stage sequencing, acceptance rules, convergence checks, time-budget management, escalation from coarse to fine search, or similar global control decisions. This section is about algorithm-level control flow, not local move mechanics and not constraint definitions. Control should remain minimal and decisive; avoid overly baroque multi-phase orchestration unless it is essential to the mechanism, and always respect the wall-clock timeout per instance. Plain language, mathematical descriptions, pseudocode-like descriptions, and numeric thresholds are allowed here. Full Python code is not allowed here.

# PARAMETERS
This section stores parameter blocks, thresholds, schedules, coefficients, ratios, penalty weights, neighborhood sizes, time fractions, and other numeric or formula-level settings that the implementation is expected to realize causally. Put genes here that make the organism's quantitative choices more specific. Parameter choices may support any earlier or later mechanism; they are not decorative comments and should not read like a late appendix. Parameter blocks should stay compact and grouped; do not explode the organism into many loosely connected constants. This section should not introduce a brand-new algorithmic idea that properly belongs in another section. Full Python code is not allowed here.
